Date: 2026-04-24

Scope: primary-source verification of claims that will inform `.planning/adr/0002-storage-engine.md`.

All crates.io metadata was fetched via the public JSON API (`https://crates.io/api/v1/crates/<name>`) and cross-referenced against the upstream GitHub repository via the `gh` CLI. Verification performed on 2026-04-24.

## Summary

- 36 claims verified ✅
- 1 claim contradicted ❌ (A1: redb is B-tree / copy-on-write, not B+tree and not mmap-free; claim was partially wrong on tree shape)

- 1 claim unverified ⬜ (A10 90-day-specific download window — crates.io exposes a 90-day "recent_downloads" counter but no time-series breakdown by day)

Note on status: some items below carry both a ✅ (for the broadly verified substance) and a 🟡/❌ marker for a specific sub-claim. The "Notes and caveats" section enumerates every row that is not a clean ✅.

## Notes and caveats

**A1 (redb architecture).** The original claim described redb as "pure-Rust embedded KV store (B+tree, mmap-free, ACID)". Pure-Rust: verified. ACID: verified. "mmap-free": verified — there is no `memmap2`/`mmap` dep in redb's runtime crate and the file backend uses plain `File` I/O + `sync_data`. "B+tree" is **wrong in strict terminology**: redb uses **copy-on-write B-trees** (per both its own README and design doc); the branch/leaf distinction in its design doc is B-tree-style, not B+tree-style (B+trees have values only at leaves with linked sibling pointers — redb doesn't describe that shape). This is a small correctness nit to fix in the ADR.

**A7 (redb unsafe footprint).** There is no `forbid(unsafe_code)`. 37 `unsafe ` token occurrences in v4.1.0 `src/`. Most of these are in the page-store / file-backend layer (expected for a storage engine doing raw bytes). A formal `cargo-geiger` run against the dep tree would give an authoritative metric; we did not run it inside the sandbox here.

**A9 (redb MVCC model).** Single-writer-multi-reader MVCC — not multi-writer. Acceptable for etcd-style workloads (etcd itself is single-writer via Raft) but worth naming explicitly in the ADR so nobody expects RocksDB-style concurrent writers.

**B5 (raft-engine recovery-time).** No published recovery-time-per-GB SLO. The blog + InfoQ numbers cover steady-state throughput and latency (40% write-I/O reduction, 20% tail-latency reduction vs. RocksDB), not recovery. The ADR should not cite a recovery-time number unless we measure it ourselves.

**B6 (raft-engine deps).** "Pure Rust" is directionally true (no RocksDB, no big C++ libs) but the crate does pull in `lz4-sys` (C) by default for compression. If the ADR cares about a fully C-free build, it should either set compression to `none` or replace `lz4-sys` (there's a pure-Rust `lz4_flex` option some users swap in).

**B7 (raft-engine unsafe).** 49 `unsafe ` occurrences on current `master` (v0.4.2 tag doesn't exist in git, but HEAD is within 3 weeks of the 0.4.2 crate publish). Not abnormal for a storage engine with mmap and manual buffer arithmetic. No `forbid(unsafe_code)`.

**C4 (sled production statement).** The maintainer explicitly tells users _not_ to use sled if reliability matters, and points them at SQLite. This should be treated as a hard disqualifier for any etcd-class system.

**F1 / F4 (bbolt).** Confirmed etcd uses bbolt (`go.etcd.io/bbolt` import). No numeric scale limit is published; the practical etcd limit is the `--quota-backend-bytes` setting which is almost always configured in the 2–8 GiB range. This is the key reason mango should not assume its storage engine needs to scale to TB-class datasets — it needs to be _fast and reliable at ~GB scale_.

**H4 (etcd fsync batching).** Etcd does batch raft proposals into single commits, which folds multiple client writes into one `fdatasync`. The exact batching thresholds live in etcd's raft/server code rather than user-facing docs; we did not pin them within the timebox. For the ADR, the important fact is that fsync _is_ batched — so the storage engine's commit path must also support batched / group-commit semantics cleanly.

## Open questions

1. **redb production users in public.** redb.org and the README don't enumerate production users. Cuprate (a Monero node, docs.rs shows heavy use) and a few smaller projects are visible consumers, but there is no "XYZ Corp runs redb at scale" reference we could cite. For ADR risk assessment, this is a concrete gap — redb is stable/maintained but has no public marquee deployment comparable to Meilisearch + heed + LMDB.

2. **raft-engine semver / stability commitment.** Still at 0.4.2 on crates.io. No documented stability policy. Active work continues on `master` (last commit 2026-03-10) but has not been released to crates.io in ~24 months. This is a publication-cadence risk for mango's lockfile hygiene. Worth opening a discussion issue on `tikv/raft-engine` asking about 1.0 plans before we lock in.

3. **redb unsafe-footprint formal number.** We counted `unsafe ` tokens (37) but did not run `cargo-geiger`. A proper geiger report, including transitive deps, would be a better citation for the ADR's "minimize unsafe surface" argument.

4. **bbolt max value size at the protocol level.** Not stated in the README. etcd's own max single-value limit is enforced above bbolt (`--max-request-bytes`, default 1.5 MiB). Confirming this is an etcd-layer limit, not a bbolt-layer limit, would strengthen the ADR's workload-profile reasoning.

5. **etcd fsync batching thresholds.** Documented at a conceptual level only. Reading `etcd-io/etcd/server/storage/wal/wal.go` would give authoritative numbers for the batching window and group-commit logic. Deferred — not critical for the ADR framing, relevant for the benchmark comparison.
