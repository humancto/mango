# bbolt oracle for mango

Reference implementation of the embedded-KV semantics mango's
`Backend` trait is expected to satisfy. Used in two modes:

- **diff** (ROADMAP:819, implemented incrementally across this PR):
  a correctness oracle. mango's Rust test harness runs identical
  proptest-generated op sequences against its redb-backed
  `Backend` and this bbolt binary, then compares committed state
  byte-for-byte. Divergence triggers either a harness bug fix or
  — per ADR 0002 §5 Tier-1 — an engine swap conversation.

- **bench** (ROADMAP:829, stubbed): a performance oracle. Same
  binary, different subcommand; the plan is to share protocol and
  process lifecycle rather than maintain two binaries.

This directory ships `main.go` + the Go module files, a pinned
bbolt version in `VERSIONS`, a reproducible build script, and Go
unit tests against the dispatch layer. The binary is NOT committed
— each contributor builds locally via `./build.sh` (see below).

## Layout

```
benches/oracles/bbolt/
├── main.go          # oracle source
├── main_test.go     # Go unit tests for dispatch logic
├── go.mod           # pinned go.etcd.io/bbolt
├── go.sum           # integrity checksums (go mod verify)
├── VERSIONS         # shell-sourceable pin
├── build.sh         # reproducible build (CGO=0 -trimpath)
├── BBOLT_QUIRKS.md  # accepted semantic deltas vs redb
├── .gitignore       # ignores built binary + *.db artifacts
└── README.md        # this file
```

## Building

From this directory (or anywhere — build.sh cd's to its own dir):

```
./build.sh
```

Produces `./bbolt-oracle`. The Rust harness finds it via the
`MANGO_BBOLT_ORACLE` env var (set by the CI workflow / test
invocation) or via a build-script default that points at this
path.

## Protocol (diff mode)

Newline-delimited JSON on stdin; newline-delimited JSON on stdout.
Strictly serial — one request, one response; the binary does NOT
pipeline. All binary payloads are base64-encoded with
`base64.StdEncoding` (no line breaks ever, regardless of input
length). This is the property the harness's line-splitter depends
on; see the inline comment on `decode64`/`encode64` when those
helpers land in commit 3.

### Framing

- Go side: `bufio.Scanner`, `scanner.Buffer(make([]byte, 1<<20), 16<<20)`.
  The default 64 KiB cap overflows on realistic `snapshot`
  responses; the 16 MiB cap is the agreed-upon ceiling.
- Rust side: `BufReader::with_capacity(16 << 20, stdout)` on child
  stdout; `BufWriter` on child stdin.
- Protocol errors (unknown op, bad JSON) return
  `{"ok": false, "error": "protocol: ..."}` and the binary keeps
  processing. Only `close` ends the process cleanly.

### Ops implemented in this commit

| Op       | Fields          | Response                                |
| -------- | --------------- | --------------------------------------- |
| `open`   | `path`, `fsync` | `{ok: true}` or `{ok: false, error}`    |
| `close`  | —               | `{ok: true}` — oracle exits after reply |
| `reopen` | —               | close + reopen at latched path/fsync    |

### Ops added in subsequent commits

`bucket`, `put`, `delete`, `delete_range`, `get`, `range`,
`snapshot`, `size`, `commit`, `commit_group`, `rollback`,
`compact`. Full op table lives in the diff harness plan
(`.planning/differential-backend-vs-bbolt.plan.md` §2) and will
move here once all ops are wired.

## Post-mortem artifacts

On divergence, the Rust harness writes
`target/differential-failures/<utc-timestamp>-<case-hash>/` with
`ops.json` (the offending sequence), `bbolt.db`, `redb.db`, and
`diff.txt` (a minimal pairwise diff). These directories are NOT
auto-cleaned because operators need them for root-cause analysis;
GC them manually once the divergence is understood and filed.

## Threat model

**Attack surface:** this binary runs as a subprocess of `cargo
test`, reading untrusted JSON from a pipe whose other end is
written by the Rust test harness. The JSON is machine-generated by
proptest, so the realistic threat is a malformed proptest output
triggering undefined behavior — not a remote attacker.

**Supply-chain posture:** the only linked third-party dep is
`go.etcd.io/bbolt` (Apache-2.0, etcd-io, CNCF project). `go.sum`
pins integrity checksums; `go mod verify` runs as part of
`build.sh`. The go.mod graph may list test-only modules
(`stretchr/testify` for bbolt's own tests) that are NOT linked
into the oracle binary — confirm with `go list -deps .`.

**Bumping bbolt.** Update `BBOLT_VERSION` in `VERSIONS`, run
`go get go.etcd.io/bbolt@<NEW>` in this directory, then
`go mod tidy`, then `go mod verify`, then commit the
go.mod/go.sum diff. CI refuses a merge if go.mod is desynced from
go.sum. The differential harness runs the thorough suite on
milestone tags, so a bad bbolt bump is caught before release.

**No other Go deps permitted.** Any future PR adding a third-party
module to `go.mod` requires an ADR revision and an explicit
supply-chain review. The point of a reference oracle is to be
trustable on inspection; dependency sprawl defeats that.
